# Project Generation Rules (프로젝트 생성 일반 규칙)

이 문서는 에이전트가 이 프로젝트에서 코드/파일을 생성할 때 항상 지켜야 하는
공통 규칙이다. `@persona-developer.md`와 `@persona-designer.md`의 관점을
아래 규칙 안에서 조율해서 적용한다.

## 언어 정책
- 모든 응답, 주석, docstring, UI 문구, 커밋 메시지는 **한국어**로 작성한다.
- 코드 식별자(변수명/함수명)는 영어를 사용하되, 설명은 한국어 주석으로 보완한다.

## 기술 스택 (고정)
- 언어: Python 3.11+
- UI 프레임워크: Streamlit
- DB: SQLite (표준 `sqlite3` 라이브러리, ORM 사용하지 않음)
- 영상 생성 API: Atlas Cloud (Seedance 2.0 엔드포인트)
- 환경변수 관리: `python-dotenv`
- 이 스택 외의 프레임워크(FastAPI, Django, PostgreSQL 등)는 사용자가 명시적으로
  요청하기 전까지 제안하지 않는다.

## 프로젝트 구조 (기본 템플릿)
```
project-root/
├── app.py                 # Streamlit 메인 엔트리포인트
├── api_client.py          # Atlas Cloud API 호출 로직 (submit/poll/download)
├── db.py                  # SQLite 초기화 및 CRUD 함수
├── config.py              # 상수, 설정값
├── .env                   # 실제 API 키 (git 제외)
├── .env.example           # 키 없이 형식만 예시로 제공
├── .gitignore
├── requirements.txt
├── history.db             # 생성 이력 (런타임 생성)
├── outputs/                # 다운로드된 영상 저장 폴더 (런타임 생성)
└── README.md
```
- 파일을 새로 만들 때는 이 구조를 벗어나지 않는다. 벗어나야 한다면 먼저 사용자에게 이유를 설명한다.

## 파일 생성/수정 규칙
- 새 파일을 만들기 전, 기존 구조에 이미 적합한 파일이 있는지 먼저 확인한다.
- 하나의 파일에 UI 로직과 API 호출 로직을 섞지 않는다 (`app.py`는 UI만, `api_client.py`는 통신만).
- 민감 정보(API 키, 개인 파일 경로)가 포함된 예시 코드나 로그를 만들지 않는다.
- 모든 함수는 실패 케이스(네트워크 오류, 타임아웃, 잔액 부족)를 처리한 뒤에 "완료"로 간주한다.

## 개발 vs 배포 범위
- 현재 범위는 **로컬 개인 사용**으로 한정한다. 클라우드 배포, 다중 사용자 인증,
  결제 시스템 관련 코드는 지금 단계에서 생성하지 않는다.
- 사용자가 "이제 배포해줘" 또는 "다른 사람도 쓸 수 있게 해줘"라고 명시적으로 요청하기 전까지는
  이 범위를 유지한다.

## 콘텐츠/정책 관련 주의사항
- 실인물 얼굴을 참조 이미지로 사용하는 기능을 구현할 때, Seedance 2.0 모델 자체의
  콘텐츠 정책(실사 얼굴 직접 업로드 제한 가능성)을 코드 주석과 에러 처리에 반영한다.
- API 제공처의 이용약관(리셀/재배포 제한 등)에 위배될 수 있는 기능은 구현 전
  사용자에게 확인을 요청한다.

## 작업 진행 방식
1. 새로운 기능 요청을 받으면, 먼저 Implementation Plan(구현 계획)을 간단히 정리해서 보여준다.
2. 사용자 승인 후 코드를 생성한다.
3. 생성 후에는 실행 방법(예: `streamlit run app.py`)과 필요한 사전 준비(API 키 발급 등)를 안내한다.
4. 큰 변경사항은 한 번에 다 만들지 않고 단계별로 나눠서 진행 상황을 보고한다.
